# LTX-Desktop 架构探索方案总结

是的，你的直觉非常准确。经过我们这几轮的拉扯和排错，不管触发了多少个看起来五花八门的红字报错，本质上我们**自始至终只在解决两件大事情**：
1. **绕过显卡和环境验证，在任意机器启动前端 UI**。
2. **解决独立的前后端（Windows 本地 vs Linux 云端）之间底层文件系统、协议与网络路径的割裂问题**。

为了理清思路，我将我们现有的做法（方案一）和接下来我强烈建议采取的新做法（方案二）进行了单独总结。

---

## 方案一：当前方案总结 (纯前端直连云端修改补丁)

**核心思路**：彻底抛弃本地 Python 后端，让 Windows 上的 Electron 前端直接通过网络（HTTP）与远端搭载 GPU 的 Python 服务通信。

### 1. 绕过显卡验证启动前端（注入环境变量拦截 Electron 本地唤起机制）：
*   **技术细节与原理**：原本前端 UI（Electron Main 进程）在启动时，默认会执行 `electron/python-backend.ts` 内的 `startPythonBackend()`。该函数会用本地 Python 脚本拉起 `backend/ltx2_server.py`，并疯狂轮询检测其存活状态。一旦本地没有昂贵的显卡和庞大的 Torch 环境，Python 启动报错，UI 就会永久卡死在 Loading 界面。
*   **我们的修改方案**：我们发现原代码中存在一处环境变量后门。通过编写 `start-client.ps1` 脚本在启动 Electron 前注入系统环境变量 `$env:LTX_EXTERNAL_BACKEND_URL`，使得 `startPythonBackend()` 提前拦截生效。它直接将 `backendOwnership` 设置为 `'external'`，并调用 `publishBackendHealthStatus({ status: 'alive' })`，不仅强行中断了本地沉重的 Python 进程的孵化，还完美欺骗了 UI 渲染层（使其误以为服务已启动），一举剔除了前端对任何显卡硬件的束缚，做到了“能在轻薄本上秒开”。

### 2. 处理跨系统的文件系统问题：
*   **上传端限制与转换**：原版 LTX-Desktop 是给本地后端发送文件的 Windows 绝对路径（`C:\...`）让后端自己读。我们将其修改为：前端只要检测到是外网连接，就在浏览器内部把图片/音频强制读取转换为 `Blob` 字节流，然后调用远程 `/api/upload` 接口临时保存。
*   **拖拽安全的痛点**：由于 Chromium 浏览器的沙箱安全机制，只要你是通过拖拽或是非标准弹窗导入的文件，大概率会丢失绝对路径，只剩下类似 `./图片.jpg` 的伪路径。这导致前端内部转换代码大量找错文件，逼得你只能把图片拷到 LTX-Desktop 根目录才能勉强读到。
*   **回显端播放痛点**：远端生成完成后返回的是类似 `/cloud/cloud-ssd1/LTX-...` 的 Linux 路径。我们在前端打了补丁，让前端将它强制拼接为 `http://117.50.../api/media?path=...`，同时在后端放宽了 CORS 跨域策略，企图用 HTTP 强行拉取流媒体。
*   **视频编辑器的灾难（阻碍）**：这种强行拼接 HTTP 的做法在这类普通生成还能凑合，但对于 LTX-Desktop 自带的“剪辑编辑器”是毁灭性的。如果你拥有 4 条时间相近的轨道片段来回拖拽，HTTP 请求流媒体由于缺少完备的视频缓冲流头，必定会导致浏览器端严重黑屏和卡死。

## 方案二：新方案技术路线 (本地 Python 透明代理 Proxy)

**核心思路**：**回滚前端文件关于网络方向的所有改动**，让前端依然“愚蠢”地以为自己就是在和本机的原生 Python 服务器（`127.0.0.1:8000`）高频对话。我们在本地用极少量的代码写一个 **轻量级的 Python 代理（Proxy 转发器）**，完美接管文件差异。

### 1. 绕过显卡验证（轻量级反向代理服务器剥离 Torch 依赖）：
*   **运行机制**：在 Proxy 方案中，我们要彻底欺骗 Electron 前端，所以不能再注入 `LTX_EXTERNAL_BACKEND_URL` 环境变量。我们会让 Electron 像官方原版一样，踏踏实实地去执行 `python backend/ltx2_server.py` 启动本地后端。
*   **如何剥离显卡检测**：为了保证这段原生的本地 Python 能够在没有显卡或缺少 CUDA 开发库的机器上（如普通办公本）平稳启动，我们需要重写或配置 `backend/ltx2_server.py` 与 `app_factory.py`。通过新增一个命令行参数（如 `--proxy-to=http://117...`），在服务初始化阶段彻底“掐断”对 `diffusion`、`torch` 库及各种大容量模型的 `import`。将其降级为一个纯净的、瞬间启动的异步（FastAPI）转发代理端。
*   **结果**：对 Electron 而言，本地 8000 端口完美按照官方标准存活了，它放行了 UI；对运行机器而言，不需要哪怕 1MB 的显卡显存；这就在保留了原生通讯接口的同时，达成了环境与算力的终极解耦。

### 2. 处理文件系统的问题（按需触发的“镜像同步隧道”）：
*   **什么是透明镜像同步**：代理的核心思想是在你本地 Windows 和远程 Linux 之间建立一个“按需同步的镜像传输带”。前端全程盯着本地的 `C:\...\outputs\` 文件夹（镜像柜子），对底层的搬运过程**零感知**。
*   **上传镜像（本地到云端）**：前端选定了一张本地图片（如 `C:\图片\xx.jpg`），并将其路径通过 JSON 发给代理。由于代理是用 Python 跑在本地的，它能毫无阻碍地使用系统权限读取这个文件，并在后台光速将其“推送镜像”到云端服务器，接着把 JSON 里原有的本地路径悄悄替换成云端刚刚返回的 Linux 路径，再把请求放行给云端。
*   **下载镜像（云端回本地）**：当远端 GPU 算力生成完视频（如 `/root/.../outputs/video.mp4`），代理会将其拦截，在后台自动把视频拉回本地的 `C:\...\outputs\` 镜像文件夹。最后，代理给前端返回一条修改过的成功消息：“嘿，生成好了，文件就在本地的 `C:\...\outputs\video.mp4`”。
*   **获得极致性能体验与兼容**：前端由于收到的是原生 Windows 绝对路径，它会按照设计初衷，使用 `file://C:/...` 进行原生的全性能快速本地读取。无论是反复拖拽播放条，还是放入 Video Editor 视频轨道进行复杂的切片剪辑，都能享受到本地 SSD 的秒级响应。黑屏、找错文件、网络缓冲慢等跨系统排异反应将彻底消失。

### 具体涉及修改和操作的文件：

1.  **回滚前端与后端的破坏性代码**：回滚 `frontend/hooks/` 和 `Playground.tsx/GenSpace.tsx` 用于强行 HTTP 流媒体的打补丁操作。还原 `electron/` 相关检测机制。
2.  **创建文件 `backend/proxy_server.py`**：这是所有的核心。它包含一个通过 `httpx` 构建的反向代理路由，它主要负责两件事处理：拦截特定路由进行文件同步，以及无脑中转其他常规的文本 API 请求到 `http://117.50.248.201:8000`。
3.  **配置接入点**：在 `backend/ltx2_server.py`（启动入口）引入判断环境机制，比如启动命令如果是单纯的 `python ltx2_server.py --proxy=117.50:8000`，它会跳过 `pipelines` 下载或启动，直接拉起代理 Server 即跑通整个 LTX。
