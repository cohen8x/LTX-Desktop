# LTX-Desktop 架构重构终极蓝图：透明代理方案

> **写给新会话的重构 AI 的话：**
> 本文档是上一轮会话经过深度源码剖析后得出的**唯一且绝对正确**的重构路线。请你仔细阅读本方案的每一个字，不要再试图去重构前端（Frontend），也不要试图去寻找原项目里有没有自带的“直连云端”配置（因为根本没有，且原版的 `adopted` 模式仅仅是清理僵尸进程的刺客代码）。请严格按照本《透明代理方案》的步骤执行代码编写。

---

## 1. 核心业务痛点与重构目标
**背景**：原版的 LTX-Desktop 是一个极其笨重的本地桌面应用。前端（React + TypeScript）和底层的基于 FastAPI 的大模型服务（加载 Torch、CUDA）被以极高的耦合度塞在了同一台 Windows 电脑里。
**目标**：把吃显卡的 AI 算力服务迁移到**自建的远端高配 Linux 节点**上，同时在本地轻薄本（无显卡）上维持前端的流畅运行。

## 2. 为什么绝对不能“修改前端直接向云端发请求”？
因为原项目是一套**“披着网页壳子的强桌面应用”**：
1. **上行发送绝对路径（打不开）**：前端发出的 JSON 里全都是 `{"imagePath": "C:\\videos\\dog.png"}`。直接发给你的 Linux 云端，云端的 Python 直接报 `FileNotFoundError` 崩溃。
2. **下行依赖本地磁盘极速读取（核心断点）**：前端带有极度复杂的 Timeline 高级视频剪辑器。它不支持通过网络流（HTTP Stream）播放视频！它强制要求后端返回的必须是本地绝对路径（如 `C:\\outputs\\123.mp4`），前端代码会拼凑成 `file://C:/outputs/123.mp4` 交给 Chromium 内核去实现零延迟的拖拽和帧率读取。如果是云端路径，编辑器直接黑屏瘫痪。

---

## 3. 终极解法：本地透明代理架构 (Transparent Proxy)
**核心思想（Zero Frontend Modification）：**
前端一字不改。我们在本地 Windows 上启动一个只有几百行的纯 Python 脚本（`proxy_server.py`）替代理原先庞大的 `ltx2_server.py`。代理拦截通信，做“文件走私”，把跨操作系统的路径网闸完全抹平。

### 代理层的四大核心职责 (给 AI 的行动指南)

#### 3.1 环境秒过与存活放行 (Bypass & Mock)
把所有发往 `/api/health` 和 `/api/models/status` 的心跳请求直接拦截。在代理内部以硬编码的方式瞬间返回 200 OK 和伪造的“显卡/模型已就绪” JSON。
**收益**：绕过前端所有的下载模型前置弹窗和 GPU 检测拦截，秒进主界面。

#### 3.2 常规路由透传 (HTTP Relay)
除文件路由外的统统放行。像 `/api/settings`，`/api/runtime-policy` 甚至 `/api/generation/progress`（进度条），代理使用 `httpx.AsyncClient` 完整接受入参并转发到远端 Linux 节点，然后原样弹回。
**优势**：极大地保留了后端的业务处理能力，例如后端的**混合计算模式 (Hybrid Computing)**将继续在 Linux 节点上生效。

#### 3.3 核心拦截之上行倒卖 (Uplink Hijack)
**目标路由**：`POST /api/generate`, `POST /api/retake`, `POST /api/ic-lora/generate` 等。
**逻辑**：
1. 拦截 JSON 请求，嗅探是否存在 `imagePath`, `audioPath`, `video_path` 等 Windows 物理路径。
2. 代理利用自身在本地的优势，执行 `open("C:\...", "rb")` 读取文件。
3. 动态通过 Form-Data 提交至远端服务器的 `[云端 IP]/upload` 临时接口（*注：这需要在远端 Linux 的 FastAPI 补充一个小路由*）。
4. 拿到云端返回的远端 Linux 地址（如 `/cloud/ssd/dog.png`）。
5. 篡改原始 JSON 载荷，替换掉 C 盘路径，将新 JSON 扔给远端正式生成接口。

#### 3.4 核心拦截之下行落盘 (Downlink Mirror)
**目标行为**：等待远端生成视频成功的 Response。
**逻辑**：
1. 拦截云端返回的成功 JSON，嗅探其中的 `video_path`（形如 `/cloud/ssd/123.mp4`）。
2. 发起极速下载，将视频立刻无损同步拉取到本地的约定缓冲目录，例如项目根目录的 `outputs/` 下。
3. 篡改下行 JSON，把 Linux 路径抹掉，写成本机下载后的物理路径 `{"video_path": "C:\\....\\outputs\\123.mp4"}`。
4. 放行给前端。前端的 React 钩子将快乐地构建 `file://` 来秒开视频。

## 4. 重构实施分步计划 Phase 1 (供新会话 AI 执行)

### Step 1: 编写轻量级本地代理服务器 (`backend/proxy_server.py`)
- 使用 `FastAPI` + `uvicorn` + `httpx`。
- 实现 `/api/health` 伪造。
- 实现请求体全量解析与路径拦截器。
- 配置灵活的超时时间（因为视频生成请求是 Long-Polling 阻塞型长连接，可能要等待好几分钟，`httpx` timeout 必须设置为 None 或极长）。

### Step 2: 为云端远端补充一个极简文件上传接口
- 在 `backend/` 的某个通用路由下帮远端的 Linux 增加一个 `/api/proxy-upload`。
- 因为原版代码默认素材都在本地硬盘，没有从 HTTP 请求体接管文件的逻辑，我们需要让代理有地方放倒卖过去的素材。

### Step 3: 修改 Electron 启动指向 (`electron/python-backend.ts`)
- 将原本 `pythonProcess = spawn(pythonPath, ["-u", mainPy])` 中指向 `ltx2_server.py` 的地方，修改为指向新写的 `proxy_server.py`。
- 将由于本地不再需要庞大的环境而被剔除的无用配置（如环境变量里的 PyTorch 配置）精简。

### Step 4: 测试与联调
- 在本地跑 `pnpm dev`。
- 观察终端输出，确保前端传来的 `C:\` 路径成功变为了 Linux 路径，并在视频生成结束后成功于本地 `outputs\` 里看到了 `.mp4` 文件。

> **结束语**：此方案保留了 LTX 原作者的所有心血结晶，通过在进程边界外实施黑客级的降维打击，实现了零瑕疵的环境解耦。请下个会话的工程师/重构 AI 带着敬畏与激动立刻开展 Step 1 到 4 的代码实施。
