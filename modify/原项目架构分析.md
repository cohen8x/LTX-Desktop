# 原项目架构分析 (深度源码级)

## 1. 真实的全貌：前端重度依赖本地物理磁盘

为了回应您的严谨质询，我刚刚对整个 `frontend/` 和 `backend/` 的源码和全部路由（`_routes/`）进行了深度 `grep` 检索和源码级阅读。

原项目的架构可以被定性为：**“披着 B/S 架构外皮的 C/S 桌面级软件”**。

### 核心论据一：前端使用所有的 `file://` 协议读取视频，拒绝网络推流
在前端代码中，我发现了大量且密集的物理路径硬编码转换。例如在视频生成 Hook (`frontend/hooks/use-generation.ts`) 和核心编辑器组件中：
```typescript
// use-generation.ts 第 222-224 行：
// Convert Windows path to proper file:// URL
const fileUrl = videoPathNormalized.startsWith('/') ? `file://${videoPathNormalized}` : `file:///${videoPathNormalized}`

// views/VideoEditor.tsx 第 1467 行：
// Returns a file:// URL (videos: extracted via ffmpeg, images: already on disk)
```
**这意味着：** 前端的 `<video>` 或 `<canvas>` 渲染器，**根本不支持从网络 URL (http://...) 缓冲播放视频**。它要求后端返回的必须是**本地硬盘上的绝对路径**，随后前端会自己拼接成 `file://C:/...` 去让 Chromium 内核直接读取物理磁盘。由于 Chromium 对 `file://` 拥有本机的最高读取权限和零延迟寻址，这才实现了 Timeline 编辑器的丝滑体验。

### 核心论据二：后端的通信全权建立在绝对路径上
查看后端 `backend/api_types.py`，所有的 Request 与 Response 模型如出一辙：
*   `GenerateVideoRequest(BaseModel)` 包含 `imagePath: str`
*   `GenerateVideoResponse(BaseModel)` 包含 `video_path: str`
前端在触发生成或编辑请求时，直接通过 `window.electronAPI.readLocalFile` 等操作拿到本机的 `C:\user\images\cat.jpg`，然后原封不动地通过 HTTP POST 扔给 `localhost:8000`。

## 2. 关于 "managed" 和 "adopted" 模式的源码剖析 (再确认)

您提到的 `adopted` 模式，我也再次核实了 `electron/python-backend.ts` 中的接管逻辑。
源码证据：
1. `backendFetch` (`frontend/lib/backend.ts`) 中调用的 URL 是底层 Electron 注入的。
2. Electron 在判断端口冲突时 (`python-backend.ts` 第 348 行)：
   `const explicitUrl = 'http://127.0.0.1:${process.env.LTX_PORT}'` 
   **这行代码是死死焊在 127.0.0.1 上的。**
3. 然后它做了什么？它发送了 `curl -X POST http://127.0.0.1:8000/api/system/shutdown`。

**不可辩驳的结论**：`adopted` 绝非远程连接模式。它是 Electron 发现每次重启桌面端、但上次的后端的 Python 进程还没死透占用了端口时，专门写的一个“刺客”逻辑：连上它、杀掉它、然后本地重新起一个 `managed` 模式。
