# LTX-Desktop 新方案技术路线：本地环境剥离与双向透明文件代理

## 1. 核心目标 (Core Objectives)

我们的终极解耦方案旨在解决跨物理机（Windows轻薄本作为控制端，Linux大显卡服务器作为算力端）运行原版 LTX-Desktop 时的两大底层限制：

1. **环境与算力强制解耦 (GPU Bypass):** 
   使前端（Electron UI）能在**任何没有独立显卡、不安装 CUDA 和庞大 Torch 环境**的办公电脑上顺利启动，同时不触发前端因“后端掉线”而永久卡死在 Loading 界面。
2. **原生文件系统透明兼容 (Native Electron Compatibility):** 
   维持 LTX-Desktop 庞大复杂的视频编辑器（Timeline Editor）的原生读写性能。通过构建“文件镜像隧道”，骗过 Electron 沙箱，使其认为所有多媒体读写依然发生在本机物理硬盘上。

---

## 2. 旧方案痛点与技术妥协回顾 (Why Front-end Patching Failed)

在我们上一轮尝试的纯前端直连方案中，直接让 Windows UI 用内网 IP 向 Linux 请求数据，产生了不可逾越的技术壁垒：
1. **拖拽安全性（沙箱隔离）**：浏览器阻止前端脚本随意抽取操作系统的绝对路径（Drop 事件中获取到的会是残缺的 `./图片.jpg`）。我们被迫用 Blob 强转重构上传逻辑。
2. **黑屏与 `Range` 请求特性缺失**：当大体量的视频编辑器试图去“剪切”、“快进”一段通过 `http://` 播放的云端流媒体视频时。往往由于异地服务器未设置严格的跨域规则、或者是并发请求丢失媒体头（例如 `Accept-Ranges: bytes`），最后导致播放器在缓冲时彻底卡死或变黑。此时前端彻底失去了在本地 SSD 读写文件（`file://C:/...`）所享受的零延迟快进特性。

因此，**修改前端代码永远只是在打补丁。唯一的出路是重组并欺骗通信管道**。

## 3. 具体技术实现细节 (Detailed Technical Implementation)

新方案是一套由本地“无模型空壳 Python”与远端“重装 GPU 节点”配合组成的 **“中间人代理架构 (Man-in-the-Middle Proxy)”**。

### 3.1 零依赖环境的“代理启动” (绕过显卡验证)

**原版痛点：**
Electron 原版代码的 `main.ts` 会去本地直接执行 `python backend/ltx2_server.py`。原版的 `ltx2_server.py` 在初始化阶段无论三七二十一，就会直接触发 `torch`, `diffusers` 等庞然大物的导入。此时如果 Windows 没有显卡环境，瞬间报 `DLL Load Failed` 或 `No CUDA device` 并退出。紧接着，Electron 收不到 8000 端口的 Health 信号，应用卡死在 Loading 屏。

**技术实现：**
1. **条件导入 (Conditional Import) 机制**：修改 `backend/ltx2_server.py` 及关键构建工厂 `app_factory.py`，新增 `--proxy-target=http://117.50.248.201:8000` 启动参数。
2. **隔离大模型路由挂载**：在配置中检测到代理模式后，服务端将跳过所有 `Pipelines` 类的构建（意味着 `torch` 全家桶根本不会被导入，显存占用为 0）。
3. **保留生命健康接口**：依旧在本地 `127.0.0.1:8000` 提供原始的 FastAPI 服务。它能瞬间通过前端的 `/api/health` 发来的心跳侦测（执行耗时缩减至不足 0.5秒）。
4. **结果**：前端应用将被完美欺骗，正常启动全功能 UI 界面。

### 3.2 拦截与镜像文件的双向代理链路 (Proxy Traffic & Mirroring)

不改前端代码，前端仍然发送 `{"image_path": "C:\\用户\\图片\\cat.jpg"}` 请求给本地 8000 端口。此时的 8000 端口（我们的本地空壳 Python）接管并调度这股流量：

#### 步骤一：上传截获与推送 (Payload Interception - Local to Remote)
*   **截获**：代理服务器侦测到前端发来了 `/api/generate` 且携带有 `image_path`（且这是一个本机真实的、未受浏览器污染的绝对路径）。
*   **同步云端**：代理服务器（拥有系统的普通用户最高权限）直接使用 `pathlib` 读取这张 C 盘上的 `cat.jpg`，并在后台将其作为文件表单 POST 到远端服务器 `http://117.50.../api/upload`。
*   **路径重构**：收到远端发来的 Linux 物理存盘路径（例：`/root/prj/.../uploaded_cat.jpg`）后，代理服务器篡改（Replace）原始的 JSON payload，并再次放行，转发给远端真正的算力服 `POST /api/generate`。

#### 步骤二：轮询与状态透传 (Transparent Polling)
*   代理只需原样转发（Passthrough）前端对 `/api/generation/progress` 的拉取请求。

#### 步骤三：极速下载与虚假返回 (Download Hook - Remote to Local)
这是**彻底解决视频黑屏、播放器卡死、无法多轨修剪**的最关键一步。
*   当远端 GPU 在近五分钟计算完毕后，返回数据给代理：`status: complete, video_path: /root/prj/.../output_123.mp4`。
*   代理端 **暂扣此响应**，不急于还给前端。它首先对远程发起高带宽下载请求 `GET /api/media?path=...`。
*   下载流瞬间写入本地专属的 `C:\prj\...\LTX-Desktop\outputs\output_123.mp4`。
*   文件落地完毕后，代理将原 JSON 改写为：`{"status": "complete", "video_path": "C:\\prj\\...\\outputs\\output_123.mp4"}`。
*   最后，将这条携带 Windows 原生盘符的响应回放给等待中的 Electron。

#### 步骤四：原生满血运行 (Zero-Latency Local Serving)
前端在获取到回溯的本地路径后，使用 `file:///C:/...` 发放给 `<video>` 组件和 Timeline 编辑器。
从此，所有视频拖进度条和编辑回放，都在**本地固态硬盘**上闭环进行，网络彻底只用于“上传素材”和“下载成品大文件”，将复杂的大型网络串流逻辑被硬核降维打击抹除掉。

## 4. 后续行动方案预告
在此明确思路蓝图后，我们下一步实施将聚焦于：
1. **全面还原前端钩子**：将 `use-generation.ts` 及所有组件中为了应付异地流播放打的“强制 Blob 上传与 HTTP 补丁”完全撤销，让业务逻辑保持干净。
2. **编写路由拦截器（Proxy Server）**：在本地 Python 增写透明转化模块和拦截脚本，完成上述四步同步流程。
